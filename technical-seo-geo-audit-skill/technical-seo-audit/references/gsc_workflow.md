# GSC / GA4 / PageSpeed 半自动检查流程

本流程用于那些无法仅靠公开抓取可靠判断的 SEO 检查项。涉及登录、私有数据、提交 Sitemap、请求索引、关联属性或修改配置时，必须先获得用户明确授权，并在需要用户登录时让用户接管浏览器或提供导出文件。

所有结论必须记录数据日期、属性名称、设备、国家/地区、筛选条件和样本范围。Search Console 数据存在延迟和抽样，不能把单日波动直接判断为长期问题。

## 1. 需要用户提供或授权的数据

| 数据 | 最佳格式 | 目的 |
| --- | --- | --- |
| GSC 页面索引报告 | CSV 导出或截图 | 识别“已抓取未索引”“已发现未索引”“重复网页”“备用页”等索引问题。 |
| GSC Sitemap 报告 | CSV 或截图 | 验证提交状态、已发现 URL 数、读取错误。 |
| GSC Core Web Vitals | 报告截图或 URL 组导出 | 识别移动端/桌面端 LCP、INP、CLS 问题模板。 |
| GSC 安全与人工操作 | 截图 | 排查安全问题和人工处罚。 |
| URL Inspection 结果 | 截图或复制文本 | 核心页面逐页确认 Google 选择的 canonical、抓取状态、渲染状态。 |
| GA4 流量与转化 | CSV/截图 | 判断 SEO 问题的业务影响和优先级。 |
| GTM 容器配置 | 预览结果或导出 | 检查事件、转化、重复触发、Consent Mode。 |
| PageSpeed Insights / CrUX | URL 或 API/截图 | 获取真实用户体验和 Lighthouse 建议。 |
| 服务器日志/CDN 日志 | CSV、日志样本或摘要 | 验证 Googlebot/Bingbot 抓取频率、状态码、模板异常和 crawl budget 浪费。 |

## 2. GSC 检查步骤

第一步确认属性类型。优先使用 Domain Property，因为它覆盖 http、https、www、non-www 和子域；如果只有 URL-prefix 属性，需要检查所有可访问版本是否均已验证，并确认规范版本的数据是否完整。

第二步检查 Sitemap。进入 Sitemap 报告，确认 sitemap index 或 sitemap.xml 已提交，状态为成功，已发现 URL 数与预期接近。若“无法读取”“抓取失败”“包含无效 URL”或 URL 数明显偏低，需要回到公开抓取脚本和 CMS 生成逻辑定位原因。

第三步检查页面索引。按问题类型导出或记录 URL，优先关注核心页面目录。对“已抓取 - 尚未编入索引”应检查内容质量、重复度、canonical、内部链接和页面价值；对“已发现 - 尚未编入索引”应检查抓取预算、服务器响应、站内链接深度和 Sitemap 更新；对“重复网页，Google 选择的规范网页不同于用户指定的规范网页”应检查 canonical、内部链接、Sitemap 和重定向信号是否一致。

第四步抽样使用 URL Inspection。每类模板至少检查 3–5 个 URL，记录 Google 能否抓取、是否允许索引、用户声明 canonical、Google 选择 canonical、上次抓取时间、抓取页面截图和资源加载问题。

第五步检查 Core Web Vitals。以 URL 组为单位处理，因为问题通常来自同一模板。当前核心指标关注 LCP、INP、CLS。LCP 优先检查首屏图片、服务器响应、渲染阻塞资源；INP 优先检查 JavaScript 主线程、第三方脚本和交互处理；CLS 优先检查图片尺寸、广告位、字体加载和动态插入组件。

第六步检查安全与人工操作。若存在安全问题或人工处罚，应把它列为 P0/P1，并暂停普通优化项，先完成清理、申诉或安全修复。

## 3. GA4 / GTM 检查步骤

GA4 检查应确认自然搜索流量、落地页、转化事件、表单提交、试用注册、购买或关键业务事件是否被正确记录。若 SEO 页面流量正常但转化异常，需检查页面体验、CTA、事件追踪和归因配置，而不应只从技术 SEO 角度下结论。

GTM 检查应确认同一事件没有重复触发，关键表单和点击事件在 Consent Mode、跨域、SPA 路由变化下仍能记录。对于 B2B/SaaS 站点，建议单独核对 demo request、signup、pricing click、docs search 等事件。

## 4. 服务器日志 / CDN 日志检查步骤

如果用户能提供日志，应抽样 Googlebot、Googlebot Smartphone、Bingbot 和其他重要爬虫的请求。检查高价值 URL 是否被抓取、是否返回 200、是否存在大量 3xx/4xx/5xx、是否浪费在参数页、搜索结果页、过滤页、重复分页或静态资源错误上。日志结论应与 Sitemap、robots、canonical、内部链接深度和 GSC 索引报告交叉验证。

## 5. PageSpeed / CrUX 检查步骤

对于核心模板，分别检查首页、产品页、价格页、博客文章页、文档页和转化页。优先使用真实用户数据判断 Core Web Vitals 是否通过；实验室数据用于定位原因。若脚本输出显示响应慢或页面资源重，应结合 PageSpeed 的机会项和诊断项定位 LCP 元素、阻塞资源、未压缩图片、未使用 JS/CSS、缓存策略和第三方脚本。

## 6. 交付格式

输出时必须把“公开抓取可确认的问题”和“需要 GSC/GA4/PageSpeed 私有数据确认的问题”分开。涉及截图或导出文件时，报告应注明数据日期、属性名称、筛选条件和样本范围，避免把临时波动误判为长期问题。
